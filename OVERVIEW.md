# VibeCode — チーム共通認識

> このプロダクトに関わる全員が最初に読む1枚。細かい仕様は `PROTOCOL.md`（プロトコルの唯一の正）、
> 設計は `DESIGN.md`、タスクは `ISSUES`（送信=Vibes / 受信=vibe-receiver）。

## 一言で何か
**スマホの振動でURLを送るプロダクト。** 人間が画面の楽譜を見て「短・長」ボタンで演奏すると、スマホが振動し、PCがそれを聞き取ってサイトを開く。**QRコードを、カメラと光の代わりに「振動」でやる。**

## なぜ面白いか（ハッカソンの軸 = 技術の無駄遣い）
人間が振動を"演奏"してURLを開くという無駄さ。その裏で、実は機械が同じ振動を読んでいる（＝QRと同じことをしている）。発表はこの**二段構え**で落とす:
1. まず人間が楽譜を演奏してURLが開く → 体験として面白い
2. 実は機械も同じ振動を読める。さらにURLそのものも振動に乗せられる → QRと同じ仕組みを振動で実装している、と技術で落とす

## 全体の流れ（これだけは全員頭に入れる）
```
URLを登録 → 番号(id)が振られる → その番号を楽譜にする
→ 人間が演奏（スマホが振動）→ PCのマイクが聞く
→ 番号を復元 → 番号からURLを引く → ブラウザで開く
```

## ★最重要の決め事：基本はURLそのものを送らない
振動は情報を運ぶ帯域が細い。URLを丸ごと送ると何十回も振る羽目になる。
だから **「URL = 42番」のように短い番号(id)だけを振動で送り**、番号とURLの対応表（Supabase）は別に持つ。これが本線の根幹。

## 2つのモード（ここが認識のキモ）
このプロダクトには**2つの送り方**があり、振動フレームの先頭1ビット（モードマーカー）で切り替わる。**最初から両対応できる設計**になっている。

| | モードA：id方式【本線】 | モードB：QR方式 / 直接符号化【拡張】 |
|---|---|---|
| 送る中身 | 短い番号(id)だけ | URLそのものを0/1に変換して丸ごと |
| 事前登録 | 要る（Supabaseに登録してid発行） | 不要・DBも不要 |
| 譜面の長さ | 短い（人間が演奏できる） | 長い（自動演奏限定） |
| 位置づけ | **手動演奏の主役・まず完成させる** | **本線完成後のstretch** |
| ひとこと | 短縮URLの振動版 | 「QRと同じ仕組みを振動で」 |

- モードマーカー = 0 なら id方式、1 なら QR方式。受信側はこの1ビットで解釈を切り替える。
- **進め方の鉄則**: モードA（本線）を完成させてから、モードB（QR方式）に着手する。同時にやると本線が崩れる。issueでは X1/X2 = stretch として本線の後ろに置いてある。

## 体験の主役は「手動演奏」
自動でも振動を出せる（確実なデモ・受信チューニングの基準用）が、メインは**人間が楽譜を見てボタンで演奏すること**。だから楽譜表示と演奏UIが一番大事な見せ場。

## 3つの部品と担当
- **送信側（Flutter / スマホ・リポジトリ: Vibes）**
  - Aさん = 振動の下回り（震わせる）
  - Bさん = 画面・楽譜・演奏UI
- **受信側（PC / Python・リポジトリ: vibe-receiver）**
  - 自分 = マイク → 復号 → URL表示
- **対応表（Supabase）** = 番号↔URL。送受信の両方から見る共通DB。

## 信号のルール（送受信で絶対に一致させる）
- 短い振動 = 0、長い振動 = 1
- id は 8ビット（0〜255）、MSB first
- フレーム先頭にプリアンブル（長い振動2連発）で「始まるよ」の合図
- その後ろにモードマーカー1ビット → ペイロード
- **詳細な数値は `PROTOCOL.md` が唯一の正。勝手に変えない。変えるなら3人合意。**

## 受け取る物理構成
ラップトップにスマホを直置き（接触・近距離）。スマホの振動を**PCの内蔵マイク**で拾うのが本線。将来はピエゾ・机越し・加速度センサーにも拡張できる構造（チャンネル差し替え式）。

## Supabase セキュリティ方針（厳守）
- アプリ（Flutter）に埋め込んでよいのは anon key のみ
- service_role key は絶対にアプリ・リポジトリに入れない（サーバー専用の管理者鍵）
- キーは `.env` など gitignore対象に置き、コミットしない
- DB側で RLS（Row Level Security）を必ず有効化する
  - `urls` テーブル: SELECT は許可（逆引きに必要）
  - UPDATE / DELETE は禁止
  - INSERT はデモ要件に応じて制限（未認証で誰でも書ける状態にしない）

## 開発ルール（最低限）
- main 直push禁止。issueごとにブランチ → PR → マージ。
- ブランチ名: `<type>/<issue番号>-<内容>`（例 `feat/R4-mic`）
- `PROTOCOL.md` と各 `config`/`constants` の定数は3人合意なしに変更しない。
