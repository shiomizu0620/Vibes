# VibeCode 通信プロトコル仕様（送受信 共通・唯一の正）

> **このファイルがプロトコル定数の唯一の正。** 送信側（Flutter / スマホ）と受信側（PC / センサー）は
> 必ずこの1ファイルに合わせる。変更したら両側の担当に必ず共有し、両実装を同時に更新する。
> CLAUDE.md や各実装に定数を二重に書かない（ここを参照する）。

## 方式
- URL本体は振動で送らない。**Supabase で各URLに数値id（整数）を割り当て**、**idだけ**を振動で送る。
- 受信側は id を復元 → DBで逆引き → URLを開く。
- 理由: 振動は帯域が細く、URL全体を送ると何十回も振るはめになるため、番号方式に確定。

## ビット表現
- 短い振動（ブッ）= `0`
- 長い振動（ブーッ）= `1`
- ビット順: **MSB first**（最上位ビットから送る）
- id ビット長: **____ ビット**　【要決定 / 推奨 10〜12】

## タイミング定数　【暫定値・チーム合意で確定】
| 要素 | 値 | 備考 |
|---|---|---|
| 短い振動 (0) | 150ms ON | モーター立ち上がりの都合で最短はこの辺 |
| 長い振動 (1) | 450ms ON | |
| ビット間 gap | 150ms OFF | |
| プリアンブル | `[400ms ON, 200ms OFF] × 2` | 同期＋テンポ基準 |
| 終端マーカー | ____ | 【要決定 / 有無と形式】 |

> 注: プリアンブルの 400ms ON は 長(450ms) と近く紛らわしい。
> 混同を避けるためプリアンブルON時間を明確に変える案も検討（例: 700ms）。【要検討】

## フレーム構成
```
[プリアンブル] → [id のビット列: MSB → LSB] → [終端マーカー(任意)]
```

## 人間／機械 両対応
- 同じ振動信号を、**人間が「短／長」ボタンで打ち返して**も、**機械が加速度センサー等で復調して**も
  正しく解釈できること。
- 復調は **プリアンブルで単位時間を正規化** してから短/長を判定する（送信側のテンポ個体差を吸収）。

## 受信環境（前提）
- 机上は振動が逸れて厳しい。**ラップトップにスマホを直置き（接触・近距離）**を基本とする。
- センサー候補: 外付けIMU（MPU6050等, 数百Hz〜1kHz, USBシリアル）/ PCマイク（44.1kHz）/ 2台目スマホIMU（100〜500Hz）。
- 検出には最短パルス中に5〜10サンプルあれば十分。150msパルスなら100Hzでも余裕。
  → **サンプリングレートよりモーター立ち上がり（最短150ms）が律速**。

## 信頼性
- 長い id は累積タイミングずれに弱い。**10〜12ビット程度に抑える**か、パリティ/チェックサムを付ける。

## 変更管理
- 本ファイルが唯一の正。送信側・受信側はここの値を各実装にコピーして一致させる。
- 変更時は3人で合意 → 両リポジトリを同時更新。
