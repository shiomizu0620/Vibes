## プロダクト概要
**バイブコード**: スマホの振動でURLを送り、受信側がそれを開くシステム。

- 送信側（**このリポジトリ** / Flutter）: 「短い振動」「長い振動」を組み合わせたパターンを出力する。
- 受信側（**別リポジトリ** / PC・Python）: 振動をマイク／加速度センサー等で検知し、復号してURLを開く。
- **人間／機械 両対応**: 同じ振動信号を、人間が「短/長」ボタンで打ち返しても、機械がセンサーで復調しても解釈できる。発表は「人間で体験 → 実は機械も読める（QRと同じ）」の二段構え。
- 言葉遊び: vibe（雰囲気）× vibration（振動）× code（コード/譜面）。

**このリポジトリは送信側のみを扱う。** 受信側のコードはここには置かない。

## 対象プラットフォーム
- Android（実機）および iOS（実機）。
- コードは Android / iOS 共通。**プラットフォーム分岐は書かない**（`vibration` パッケージが内部で吸収する）。
- iOSビルドには Mac + Xcode が必須（Windowsではビルド不可）。署名は Apple ID で設定。振動に特別な権限は不要。
- エミュレータ／シミュレータは起動するが物理的に振動しない（呼び出しは成功する）。実機で確認する。

## アーキテクチャ方針
レイヤを分離し、**符号化ロジックを `encoder` に隔離**する。これにより符号化ルールが変わっても他層を触らずに差し替えられる。

```
lib/
  constants.dart        共有定数（タイミング等。PROTOCOL.md と一致させる）
  vibrator_service.dart 生のパターン配列を再生（振動可否チェック含む）
  pattern_builder.dart  Pulse列 → 振動パターン配列（プリアンブル・gapを付与）
  encoder.dart          数値id → Pulse列（差し替え可能な層）
  main.dart             UI
```

設計ルール:
- タイミング定数は **`constants.dart` だけ**に書き、値は **`PROTOCOL.md` と一致**させる。各所にハードコードしない。
- `encoder` は他層から独立させ、いつでも差し替えられる状態を保つ。
- 振動の挙動はプラットフォームで異なるが、**送信側コードでは分岐しない**。差異は受信側のチューニングで吸収する。

## 通信プロトコル（共有仕様）
**プロトコル定数の唯一の正は `PROTOCOL.md`。** 送信側・受信側は必ずこの1ファイルに合わせる（3人で共有）。CLAUDE.md には詳細を二重に書かない。

要点のみ:
- 方式: URL本体は送らない。Supabaseで各URLに**数値id**を割り当て、**idだけ**を振動で送る。受信側はidを復元→DBで引いてURLを開く。
- ビット: 短い振動(ブッ)=0 / 長い振動(ブーッ)=1。**MSB first**。
- プリアンブル・タイミング・idビット長などの具体値は `PROTOCOL.md` 参照。

### プラットフォーム間の注意
iOSの振動（Taptic Engine）は短くキレのある振動が得意で、長い連続振動は不自然になりやすい。同じパターンでも音・感触が別物になるため、**受信側の検知閾値はプラットフォームごとに調整が必要**。デモ本番では送信側を1プラットフォームに固定するのが安全。

## ロードマップ
全体の流れ（順序）。**現在どこを作業中かは `docs/INSTRUCTIONS_*.md` を参照**すること（このファイルには現在地を書かない）。

- **M1**: 環境構築 ＋ 短/長の振動を任意の並びで出力。
- **M2**: encoder 実装（数値id → ビット列 → 短/長Pulse列、MSB first、プリアンブル付与）。`PROTOCOL.md` に従う。
- **M3**: Supabase連携（URL登録→id発行、id→URL逆引き）。受信側との結合。
- 受信側（PC/Python）は別リポジトリで並行開発。
